Most agent infrastructure conversations still focus on cognition: memory, planning, tools, and execution.

But once agents start paying each other, hiring each other, and splitting value across multi-step workflows, cognition is only half the system. The other half is economics.

That is where [ACP (Agent Commerce Protocol)](https://github.com/agentralabs/acp-protocol) is aiming: an open protocol for how autonomous agents identify, negotiate, settle, and prove economic interactions.

ACP is currently marked **Draft v0.1**, but the core framing is already clear and useful.

## The Core Claim

ACP treats agent commerce as a protocol problem, not an app problem.

Instead of each platform inventing custom payloads and opaque settlement flows, ACP defines five primitive objects:

1. `AgentIdentity`
2. `NegotiationContract`
3. `SettlementIntent`
4. `ExecutionReceipt`
5. `VerityReceipt`

If two systems can produce and consume these five objects, they can transact in a common language.

That design choice is important because it separates:

- **economic coordination** (protocol)
- **money movement rails** (implementation)

ACP explicitly does not try to be a payment processor. It defines the agreement and accountability layer above payment systems.

## The Five Primitives

### 1) `AgentIdentity`

Think of this as an economic passport for an autonomous agent. It includes who the agent is, what it can do, pricing model, SLA declarations, and reputation history.

### 2) `NegotiationContract`

This is the machine-readable deal state between agents. ACP keeps negotiation finite and strict with four states:

`OFFER -> COUNTER -> ACCEPT | REJECT`

### 3) `SettlementIntent`

This object defines escrow semantics: value lock, release condition, verification mode, split rules, and failure outcomes.

### 4) `ExecutionReceipt`

This is the economic audit record: what happened, what was paid, and whether performance matched declared terms.

### 5) `VerityReceipt`

This is the most ambitious part of ACP. It records not just outcomes, but **decision reasoning** in a replayable, signed form.

In plain terms: if a dispute occurs, ACP wants the system to show _why_ the decision happened, not just that it happened.

## One Canonical Flow

ACP keeps the macro lifecycle minimal:

`REGISTER -> NEGOTIATE -> EXECUTE -> SETTLE -> AUDIT`

This constraint is more than aesthetics. In protocol design, reducing surface area usually improves interoperability and lowers integration ambiguity.

It also makes compliance tooling more tractable later.

## Why This Matters

The current agent ecosystem has a familiar gap:

- Agents can call tools, but payment semantics are ad hoc.
- Escrow logic is inconsistent or absent.
- Multi-agent revenue splits are custom glue code.
- Disputes have no deterministic replay standard.

ACP’s model is trying to solve exactly those gaps with deterministic object boundaries.

If that succeeds, it creates a credible path for:

- cross-vendor agent contracting,
- standardized settlement intent semantics,
- auditable machine-to-machine economic trails.

## Implementation-Level Notes That Stand Out

From the current spec/docs, a few details are technically strong:

1. **Idempotency is explicit** for settlement behavior.
2. **State-machine constraints** are called out instead of left implied.
3. **Signed object discipline** is central, not optional.
4. **Schema-first posture** (JSON Schema under `acp/schemas`) makes ACP testable.

That last point is important. Protocols only become real when they can be validated mechanically.

## What Is Still Open

ACP is honest about being early, and several hard questions remain:

1. Dispute resolution rules that are deterministic and broadly acceptable.
2. Monetary policy design around ACP-native credit models.
3. Legal standing of replayable decision receipts across jurisdictions.
4. Vertical schema evolution beyond initial capability sets.

These are not minor details. They are where protocol ambition meets reality.

## My Read

ACP is best understood as a **coordination and accountability layer for agent commerce**.

Not a chain.
Not a payment app.
Not a marketplace.

A language.

And if autonomous systems are going to transact at scale, shared language with deterministic receipts is the only path that does not collapse into brittle integration silos.

If you are building agent marketplaces, service brokers, orchestration runtimes, or settlement engines, ACP is worth studying now while the schema is still malleable.
